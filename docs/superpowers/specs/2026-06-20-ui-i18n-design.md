# UI i18n Design

## Goal

Add English and Russian localization for the Big_echo frontend UI, with translations stored in JSON files.

## Scope

This change covers only the React/Vite frontend under `src/`. It does not localize Rust/Tauri backend messages, transcription or summary prompts, saved session content, generated artifacts, or provider responses. Backend strings that already arrive in the frontend as raw text may continue to appear in their original language until a backend localization pass is designed separately.

Russian remains the default language so existing users see the same UI after upgrade. English is available through a user-facing language selector in Settings.

## Approach

Use a small in-repo i18n layer instead of adding a dependency. The app only needs two JSON locales, runtime language switching, basic parameter interpolation, and tests around translated labels. A custom provider keeps the implementation easy to inspect and avoids pulling in a larger framework for a narrow requirement.

Locale files live in `src/i18n/locales/ru.json` and `src/i18n/locales/en.json`. Shared i18n code lives in `src/i18n/` and exposes:

- A `Language` type for `ru` and `en`.
- An `I18nProvider` that stores the selected language in `localStorage`.
- A `useI18n()` hook that returns `language`, `setLanguage`, and `t(key, params)`.
- A translation helper that resolves dot-separated keys and replaces `{name}` style parameters.

`AppRoot` wraps the application with `I18nProvider`. Ant Design's `ConfigProvider` receives the matching Ant Design locale so built-in controls such as empty states, modals, and form validation follow the selected app language.

## UI Coverage

The implementation should move visible frontend UI strings into JSON translations across:

- Main, Settings, Tray, and New Version pages.
- Session list, cards, filters, artifact modal, delete confirmation, audio player, Todoist export, and summary prompt modal.
- Settings sections for general, audio, transcription, Apple Speech, Brain sync, Yandex sync, and Todoist sync.
- Frontend status labels, validation messages, button titles, placeholders, tooltips, modal text, and aria labels.

Programmatic constants that are not user-visible can stay as code. Test fixture text and backend-facing provider values should not be translated.

## Language Selection

Add a compact language selector to General settings with two options: Russian and English. Changing it updates the UI immediately and persists the choice in `localStorage`. If the stored value is missing or invalid, the app falls back to Russian.

No backend setting or migration is needed because this pass is frontend-only.

## Error Handling

If a translation key is missing, `t()` returns the key itself. This makes missing coverage visible during development without crashing the app. Interpolation leaves unknown placeholders unchanged.

The locale JSON files should have the same key shape. Tests should cover this structural parity so missing English or Russian keys are caught.

## Testing

Add focused tests for the i18n layer:

- Default language is Russian.
- Stored language is respected when valid.
- Invalid stored language falls back to Russian.
- Parameter interpolation works.
- Russian and English JSON files have matching key paths.

Update affected React tests to use translated labels where assertions depend on UI text. Add one integration-level test that switches the language to English and verifies a visible Settings label changes.

Run `npm test` and `npm run build` before pushing.

## Non-Goals

- No Rust/backend localization.
- No automatic OS language detection.
- No pluralization framework.
- No remote translation loading.
- No localization of existing saved content or generated transcripts/summaries.
